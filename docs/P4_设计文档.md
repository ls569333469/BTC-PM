# P4 — 多时间框架独立评分与胜率校准机制

> **项目阶段**: 核心算法调优验证期
> **前置依赖**: P2（评分结构解耦）、P3（三轨道回测与统一状态标签）已完成并稳定运行。

本设计文档基于对历史架构（PRD、技能池、升级方案）及 P3 交付成果的全面审计，旨在彻底解决当前预测面板中暴露的**“胜率恒定 100%”**和**“各级别预测雷同”**的两大核心缺陷。

## 一、系统现状历史溯源与问题定性

### 1.1 历史设计初衷回顾
在 `项目架构与运行流程.md` 及 P3 之前的设计中，系统设计为“以 1H 级别为主导，驱动多级别推演”：
- **数据流**: 每15分钟（后改为5分钟）拉取 **168根 1H K线** + 资金费率 + 恐惧贪婪。
- **算法层**: 缠论引擎（主信号）+ 6因子模型（辅助定量信号）基于这组 1H K线产出单一评分。
- **降维推演**: `prediction.py` 内部简单地通过硬编码的百分比（如 30M: ±0.05%, 4H: ±0.25%）在 1H 的基础上扩散出 7 档目标价，并**共享同一个方向和胜率**。

### 1.2 当前面临的致命缺陷 🔴

| 缺陷编号 | 现象描述 | 核心根因溯源 | 系统性影响 |
|----------|----------|--------------|------------|
| **Bug-A** | 预测表格的“胜率”列几乎总是显示 100% | `chanlun_engine.py` 权重设计很容易达到 1.0 (100%)。更致命的是，在 `chanlun_service.py` 2C合流决策中，当 6因子分歧严重得分极低 (<30) 时，代码直接 fallback 到使用缠论的满分（`compositeWinRate = chanlunWinRate`）。 | 回测系统失效（预测过于自信），Polymarket 指南的 Edge (边际优势) 失真。 |
| **Bug-B** | 8个时间框架的预测方向、胜率、状态标签完全一致 | 在迁移到 FastAPI 后未解耦单例执行限制。Service 层仅并发拉取一次 Binance 的 `1h` 级别的源数据，算出单分后硬遍历赋值给所有 TF 级别预测。 | 无法捕捉级别嵌套（如：5m底背驰触发反转，但1d仍是下跌趋势），导致短线/超短线 PM 市场的引导毫无意义。 |

## 二、架构改造方案 (P4 Implementation)

### 2.1 修复 Bug-B：多时间框架(MTF) 的 1对1 独立数据源与降维并发

打破旧有“依赖单次 1H 数据粗暴辐射全局”的逻辑。引入**1对1精准映射聚合**模型。利用 `httpx` 的并发特性，利用 `asyncio.gather` 同时获取 8 个不用底层 interval 的源数据进行平行推演。因为 I/O 是异步的，新增 7 个并发请求几乎不增加服务端等待耗时。

#### 2.1.1 1对1精准映射策略

不再进行粗糙的“分三大组”，而是让前端展示的每一个预测级别，都有绝对匹配的底层真实 K线支撑（并向下寻找次级别进行共振）：

| 预测目标 (TF) | 真实底层 Binance Interval | 跨级别共振锚点 (Interval Nesting) | 源 K线数量 (建议)|
|--------------|-------------------------|--------------------------------|----------------|
| `5m` | `5m` | 无 (作为最小原子触发器) | 288 (过去24h) |
| `15m` | `15m` | 受 `5m` 分数平滑牵引 | 288 |
| `30m` | `30m` | 受 `15m` 分数平滑牵引 | 288 |
| `1H` | `1h` | 受 `30m` 分数平滑牵引 | 168 (过去7天) |
| `2H` | `2h` | 受 `1h` 分数平滑牵引 | 168 |
| `4H` | `4h` | 受 `1h` 分数平滑牵引 | 168 |
| `12H` | `12h` | 受 `4h` 分数平滑牵引 | 168 (过去84天)|
| `1D` | `1d` | 受 `4h` 分数平滑牵引 | 168 (过去168天)|

#### 2.1.2 改造实施点
- **`chanlun_service.py` L43 扩展**: 一次性构建 8 个 `fetchMarketPrice` 的 Promise 数组。
- **抽取独立计算闭包**: 封装私有方法 `_compute_single_tf_scoring(klines, funding, open_interest, fng)`。对拉回的 8 组 K线分别独立调用 8 次，产出 8 套绝对独立的 `[-100, 100]` 原生分数。
- **组装与共振覆盖**: 在合并结果时，利用映射字典将 $S_{sub}$ 注入到 $S_{target}$ 的计算公式中，完成“区间套”引力嵌套。

### 2.2 重构缠论引擎原生打分：构建 `[-100, 100]` 物理连续坐标系 (实施 Bug-A)

为了彻底解决胜率极易溢出到 100% 且数值呈锯齿跳变的问题，我们构建一套完全基于缠论原生要素的**空间与时间连续函数模型**，并废除基于 `If-Else` 的离散打分。

#### 2.2.1 基础坐标系映射
- **打分域**：从 `[-100, +100]` 的单维连续向量，同时表达**方向**和**动能强度**（`-100` 为空头极限，`+100` 为多头极限，`0` 为绝对横盘）。
- **免战区 (Sideways Range) 阈值**：设定绝对门槛 `[-35, +35]`，落入此区间的动能不足以支撑 Polymarket 单边下注，强制前端判定为“横盘观望” (WAIT)。

#### 2.2.2 四大原生连续引擎 (总局限 ±100分)
通过提取内部数学成分来代替死板的步进分：
1. **中枢偏离度引擎 (±30分)**：利用空间距离平滑。
   - `Z = (当前价格 - 中枢中轴) / (中枢高点 - 中枢低点)`
   - 采用 `Tanh(Z) * 30` 压缩在 `±30` 极限内。突破越远远离成本区，分数越平滑拉升；若价格掉回中枢，分数平滑归 `0`。
2. **笔动力学引擎 (±20分)**：利用价格振幅平滑。
   - 测量当前笔的实际涨跌幅相对于过去 10 根历史笔平均涨跌幅的比例。极强多头逼空得满分 `+20`，微弱蠕动得 `+2`。
3. **线段基底推力 (±20分)**：大方向稳压器。
   - 保证在剧烈震荡中大级别不翻转，最后一段向上固定 `+20`，向下 `-20`。
4. **买卖点时间衰减核 (±30分)**：利用时间半衰期平滑。
   - 提取最近一次核心 BSP。触发瞬间引爆最强动能（如 `+30`），随着触发后 K 线数量 `N` 的推移，分数呈指数衰减：`基础分 * Exp(-N / 衰减率)` 直至归 0。

全局原生分 `Raw_Score = Z + Bi + Seg + BSP`，输出 `[-100, 100]` 的丝滑曲线。

### 2.3 跨级别共振加速器 (Interval Nesting / 区间套) (实施 Bug-B)

在上一节的 `2.1` 中，系统已经将 5m、1h、4h 的 K线并发剥离了。利用缠论“区间套”理论，将小级别的连续分数 $S_{sub}$ 引入作为大级别目标分数 $S_{target}$ 的修饰项。

#### 2.3.1 数学引力模型 
当计算某个大时间框架（如 1H）的预测时，调取小级别（如 5m）的原生分数：
- **场景 A：反向阻击（踩刹车 / 防回调接盘）**
  - 条件：两者符号相反。例如 1H 大势为 `+60`，但当前 5m 小级别算得 `-70`（大涨中的小级别暴跌）。
  - 公式：$S_{final} = S_{target} + (S_{sub} \times 0.3)$ 
  - 效果：`60 + (-21) = 39` 分。原本激进的多头动能骤减，即将跌入 `35` 的观望区，防止高位接盘被套。
- **场景 B：同向共振（踩油门 / 主升浪推背）**
  - 条件：两者符号相同。例如 1H 是 `+40`，5m 也是 `+80`。
  - 公式：在不溢出 100 的非线性加速。小级别引爆大级别行情，最终分被推拉至主升浪区。

#### 2.3.2 K线结果复盘：动量加速度 (Momentum Delta)
废弃传统的 EMA 阻尼防闪烁，迎合 Polymarket "买定离手" 的刚性特性，提取上一根 K线时间戳的历史得分 `Score_Last`：
- 计算加速度：`Delta = Score_Current - Score_Last`。
- 如果当前多头动能呈现 `+45分` 时，若算得 `Delta` 高达 `+30`，代表刚才发生了爆点级别的结构突变。前端面板将增加发令枪提醒：`UP: 45 [↑ +30]`，即“不要等了，立刻建仓”。

### 2.4 重构 6 因子量化模型：并入 `[-100, 100]` 连续向量场 (实施 Bug-A 同步改动)

为了与缠论的 `[-100, 100]` 物理打分完美缝合，彻底废弃 `scoring.py` 内部基于 `If-Else`（如 J>90 给 -2.5 分）的僵硬阶跃惩罚，全部改为纯数学的线性映射：

1. **BBands 布林带偏离度 (连续函数)**
   - 公式：`Position = (price - lower) / (upper - lower)`
   - 线性映射：`Score = (0.5 - Position) * 200`。（如打到上轨 `Position=1` 得到 `-100分`，如遇下轨得到 `+100分`）。不再是分段计分。
2. **KDJ 振荡器 (连续反指)**
   - 抛弃 >80 才警报的设定，利用 J 线的绝对值。
   - 线性映射：`Score = (50 - J) / 50 * 100`。外加金叉/死叉的脉冲常量加成。
3. **MFI 资金流与 SAR 抛物线 (连续映射)**
   - 同理，将绝对值或者与价格的距离百分比百分之百地以浮点数形式平滑折算为 `[-100, 100]`。

**合流终局 (Composite Score 融合)**：
原本在 `chanlun_service` 里极其麻烦且存在逻辑缺陷的 2C 代码（“如果因子<30听缠论”等）将**被一行代码彻底消灭**！
$$ Composite\_Raw = Chanlun\_Score \times 0.6 + Factor\_Score \times 0.4 $$
两大引擎在同一维度下实现完美力矩合成，互为推力，互为刹车！

### 2.5 最终决策与 API 降维映射 (Final Output Mapping)

经过 `2.3` 节的小级别跨级共振牵引后，得到最严密的 $Final\_Score \in [-100, 100]$。为了无缝兼容前端现有的 UI 渲染结构，必须将这一维向量重新降维映射为前端能看懂的方向和胜率柱状图。

**免战横盘区过滤 (Sideways Threshold)**：
设定绝对抛弃阈值 `T = ±35分`。这是 Polymarket 二元期权的生死线，低于这根线说明多空胜率差微乎其微，极易被双杀。
  - **实战保命机制（过滤假突破/回撤）**：如果大形态向上（如+20），但底层微观的笔或 6 因子正在暴跌洗盘（如-25），总分会被合力强行拖拽进 `[-35, 35]` 的死水沼泽（变成-5分）。此时系统毫不留情输出“横盘观望” (WAIT)，死死拦住你在回调半山腰误以为是“主升浪”而接盘的手，必须等到子级别企稳拐头、总分重新冲破 +35 边界才会发车。

**转换函数：绝对值降维**：
- **`winRate` 字段**：只取 $Final\_Score$ 的绝对值，映射为 `[0, 100]`。
  - *前端解读*：代表在这个时间节点发生的单边趋势的置信度。数字越大，说明朝一个方向运行的动能越暴烈。
- **`direction` 字段**：
  - 若 $Final\_Score > +35$：强制输出 `"up"` (绿底，看多信号)
  - 若 $Final\_Score < -35$：强制输出 `"down"` (红底，看空信号)
  - 若 $-35 \le Final\_Score \le +35$：强制输出 `"sideways"` (灰底，观望信号，此时的 `winRate` 通常处于 0-35 的极其低迷状态，指示波动率极低)

**额外动量字段 (供 UI 浮窗/发令枪使用)**：
- 将 `2.3.2` 算出的加速度差值，增加至 API Payload 中（如 `"momentum_delta": "+30"`），方便前端渲染爆点徽章。

这套降维方案，不仅完全继承了 `[-100, 100]` 物理引擎的所有连续性优势，而且 0 修改兼容了前端的原生渲染逻辑，极其干净利落。

## 三、部署、测试方案与严格风控执行流 (Staged Execution Rule)

为了保证整个复杂量化的每一次代码翻新都不发生回滚灾难，本项目执行**铁腕分段风控标准**。整个写代码的落地过程将被硬切分为 6 个独立执行段。

> **⚠️ 核心最高指令 (The Golden Rule)**：
> 每完成以下任意一个“阶段(Phase)”的代码开发和功能逻辑，Agent **必须强制暂停所有开发动作**，自行执行对应的【测试方案】。
> 将测试数据的结果完整汇报给 USER 审阅。**只有当 USER 明确回复“同意/满意/过关”后，Agent 才能获批进入下一个阶段的开发。绝不允许一口气跨阶段开发。**

### 第一阶段 (Phase 1)：并发数据源改造 (Bug-B 基建)
- **开发目标**: 重写 `fetch_analysis_data`，实现 8 大时间框架的并发请求。
- **🛑 停机检查点**: 请求用户审阅日志。

### 第二阶段 (Phase 2)：缠论核心物理引擎改造 (Bug-A 解决)
- **开发目标**: 重设 `chanlun_engine.py` 为单维 `[-100, 100]` 物理连续坐标系。
- **🛑 停机检查点**: 汇报底层输出分，请求用户审阅公式结果。

### 第三阶段 (Phase 3)：6因子振荡连续化改造
- **开发目标**: 重构 `scoring.py`，改为 `[-100, 100]` 线性连续方程。
- **🛑 停机检查点**: 汇报因子极值打分策略，请求用户审阅满意度。

### 第四阶段 (Phase 4)：区间套共振、动量差与合成降维 API
- **开发目标**: 在 `chanlun_service.py` 收尾区间套、动量差、横盘屏蔽，输出最终 Payload。

- **🛑 最终结项大审**: 请求 USER 全面审查满意并签字，正式交付 P4 核心量化引擎大包！

## 四、部署与前后端验证红线

### 4.1 代码变动面 (Scope of Impact)

- **管弦乐调度**: `backend/app/services/chanlun_service.py` (并发改造与区间套核心)
- **底层量化引擎**: `backend/app/engines/chanlun_engine.py` 与 `scoring.py` 将被彻底重构为完全的连续物理方程式，废弃旧有的离散硬编码判断，从根尖上解决恒定 100 分 Bug。
- **表示层**: `frontend/` 目录中仅 `PredictionTable.tsx` 等组件进行时间框架映射和基准价 UI 向下对齐。

### 4.2 预期验收标准 (Acceptance Criteria)

1. **多重时间刻度解绑**: API `/api/chanlun/analysis` Payload 返回的 8 个 `timeframes` 的胜率显著且不再雷同。
2. **胜率降温效应**: 极度平滑的 `50%-80%` 梯队离散值。
3. **性能健康度红线**: 完整计算需在 `1200ms` 下行耗时内返回，不阻塞 Frontend 轮询。
