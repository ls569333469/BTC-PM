# 架构梳理 — 问题清单

> 边梳理架构、边记录发现的问题，后续逐一修复

## 已发现的问题

### 问题1：缠论引擎与6因子评分关系错误
- **现状**：导图中把6因子评分画成缠论引擎流水线的最后一步（串联）
- **实际**：缠论分析和6因子评分是**两个独立的评判系统**，应该是并行关系
  - 缠论引擎（主）：K线 → 笔 → 中枢 → 背驰 → 趋势 → **方向+目标价**
  - 6因子评分（辅）：MACD背驰 + 成交量 + 布林带 + 资金费率 + 恐惧贪婪 + RSI → **综合评分**
  - 两者在 `chanlun_service.py` 里合并 → 最终胜率
- **影响**：导图误导，逻辑不清晰
- **修复**：更新导图，画成并行两条线最后汇合

---

### 问题2：分析间隔(15min) vs 结算间隔(5min) 的设计约束
- **现状**：`auto_analysis` 每15分钟产出新预测，`auto_resolve` 每5分钟检查到期预测
- **说明**：`auto_resolve` 不需要新预测，它只是从DB查已到期的旧预测去结算，所以5分钟检查一次是合理的（确保30M预测能及时被发现到期）
- **真正的问题**：PM支持5分钟市场，但15分钟才产出一次预测，意味着无法支持15M以下的PM市场预测
- **建议**：如果未来要支持5M/15M PM市场，需要将分析间隔缩短，但要考虑外部API调用频率和性能
- **当前结论**：设计合理，暂时保持不变，记录为已知约束

---

### 问题3：6因子评分没有实际参与胜率合并 ‼️
- **现状**：`chanlun_service.py` 第99行 `pred["factorScores"] = factor_scores` 只是把6因子评分**挂到**prediction对象上
- **实际**：`winRate` 完全由 `prediction.py` 内部的缠论逻辑单独计算（第105行），6因子评分**从未被使用**
- **影响**：前端显示的胜率是纯缠论胜率，6因子评分形同虚设
- **修复方案**：
  1. 缠论产出独立 `chanlunWinRate`
  2. 6因子产出独立 `factorWinRate`（将composite_score转为百分比）
  3. 合并产出 `compositeWinRate`（加权平均）
  4. 三个胜率都返回给前端

### 问题4：需要独立胜率 + 可扩展信号系统

  - `compositeWinRate` — 综合胜率
- **扩展性**：未来可能增加新的信号源（如新闻情绪信号），应设计为可插拔结构

### 问题5：回测统计排序缺少5m
- **现状**：`backtest_service.py` 第236行 `tf_order` 只有30m~24h，缺少 `"5m": 0`
- **影响**：5m预测的回测统计不会按正确顺序排列
- **修复**：`tf_order` 增加 `"5m": 0`
- **状态**：待修复（简单改动）

### 问题6：回测系统只验证缠论，缺少6因子和综合的独立回测
- **现状**：回测只根据缠论的方向+目标价判定命中/未命中，6因子评分只是存档
- **应有**：3套独立回测统计
  1. **缠论回测** — 缠论预测方向+目标价 vs 实际（当前已有）
  2. **6因子回测** — 6因子评分方向 vs 实际（待增加）
  3. **综合回测** — 综合胜率预测 vs 实际命中率（待增加）
- **价值**：可以横向对比哪个系统更准，调整权重
- **关联**：依赖问题3（6因子需先产出独立胜率）和问题4（独立胜率字段）
- **状态**：待实施

### 问题7：投注指南胜率字段命名混乱，需全局统一
- **现状**：
  - `winRate` — 表面叫"胜率"，实际是纯缠论
  - `aboveProb` — 表面叫"看涨概率"，实际也是缠论方向+缠论胜率算出来的
  - `guide_win_rate` — 缠论和PM简单平均，6因子完全没参与
- **需统一为**：
  - `chanlunWinRate` — 缠论独立胜率
  - `factorWinRate` — 6因子独立胜率（新增）
  - `compositeWinRate` — 综合胜率（缠论+6因子加权）
  - `aboveProb` — 用综合胜率计算，不只用缠论
  - `marketUpProb` — PM看涨概率（不变）
- **关联**：问题3/4/6
- **状态**：待实施

### 问题8：非缠论指标污染缠论引擎 ‼️
- **现状**：`prediction.py` 第56-83行混入了4个非缠论指标作为 bias 调整方向评分
  - `funding_bias` — 资金费率 ← **不属于缠论**
  - `sentiment_bias` — 恐惧贪婪 ← **不属于缠论**  
  - `rsi_bias` — RSI ← **不属于缠论**
  - `div_bias` — MACD背驰 ← 缠论有背驰概念但用的是MACD实现
- **后果**：
  1. 缠论不纯粹，胜率不是真正的缠论胜率
  2. 和6因子存在**4项重叠**（MACD/RSI/资金费率/恐惧贪婪），合并后会双重计算
  3. 回测结果无法反映缠论本身的准确性
- **修复**：将 `prediction.py` 中所有非缠论 bias 移除，缠论只做：笔→中枢→背驰→趋势→方向+目标价
- **状态**：待修复（中等改动）

### 问题9：缠论引擎实现不完整（约覆盖20-25%）‼️
- **当前已实现**：

| 模块 | 文件 | 实现程度 |
|------|------|---------|
| 笔识别 | `bi.py` | ⚠️ 非标准 — 5根K线窗口找极值，**跳过了分型步骤** |
| 中枢构建 | `zhongshu.py` | ⚠️ 基础版 — 4笔点重叠，缺少走势类型分类 |
| 背驰检测 | `divergence.py` | ⚠️ 简化版 — 只比较极值点MACD值，不是面积比较 |
| 趋势判断 | `trend.py` | ⚠️ 简化版 — 只看最后一笔+中枢位置 |

- **缺失的核心缠论概念（经互联网验证）**：

| 缺失项 | 标准定义 | 重要性 |
|--------|---------|--------|
| **分型(顶/底)** | 3根K线组成+包含关系合并 — 笔的前置步骤 | 🔴 高 |
| **K线包含处理** | 两根K线有包含关系时需合并为一根 | 🔴 高 |
| **线段** | ≥3笔组合，更高级别结构 | 🔴 高 |
| **走势类型** | 上涨=≥2个同向中枢，盘整=1个中枢 | 🔴 高 |
| **买卖点(一/二/三类)** | 一类=背驰点，二类=回调确认，三类=中枢突破不回 | 🔴 高 |
| **多级别递归** | 5分笔→30分线段→日线中枢 | 🔴 高 |
| **区间套** | 多级别同时确认 | 🟡 中 |
| **力度比较** | MACD面积比较而非histogram点值 | 🟡 中 |

- **结论**：当前覆盖约**20-25%**，最基础的分型都没有，笔的识别不标准，核心产出（买卖点）完全缺失
- **升级方案 — 成熟Python缠论库（经互联网调研）**：

| 库名 | GitHub | 特点 | 推荐度 |
|------|--------|------|--------|
| **chan.py** (Vespa314) | `Vespa314/chan.py` | 完整缠论框架：分型→笔→线段→中枢→买卖点，支持多级别、区间套，可视化好 | ⭐⭐⭐ 最推荐 |
| **czsc** (waditu) | `waditu/czsc` | 量化交易导向，自动分型+笔识别，多级别决策，含回测框架 | ⭐⭐⭐ |
| **chanlun** (MrMeng) | `MrMeng-hub/chanlun` | 全功能包：缠论K线/分型/笔/线段/中枢/背驰/买卖点，支持加密货币 | ⭐⭐ |
| **chanlun.py** (YuYuKunKun) | `YuYuKunKun/chanlun.py` | 轻量级，笔/线段/中枢识别，有TradingView前端 | ⭐⭐ |

- **建议**：用 `chan.py` 或 `czsc` 替换当前的 `bi.py`/`zhongshu.py`/`divergence.py`/`trend.py`
- **状态**：待大幅升级
