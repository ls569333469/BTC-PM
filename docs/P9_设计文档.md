# Phase 9: UI 彻底解耦与双参照系 (Dual-Reference) 升级方案 (P9 实施案)

## 🎯 业务目标与背景 (Summary)

随着长周期预测（如 12H，1D）的引入，系统中出现了一个极其尖锐的认知断层：
* **Polymarket 盘口** 在日线开盘时固定了基准价（如 `$69,838`），与当前狂飙的现货价形成巨大价差（护城河）。
* **底层量化引擎** 死守当前现货（如 `$71,400`）运行 K 线切片，当它判断短线要从高位回调到 `$71,333` 时，输出了准确的负数信号（如缠论 `-26`）。
* **冲突呈现**：用户在 UI 上同时看到了 `<kbd>+2.14% (大涨稳赢)</kbd>` 的盈亏比，旁边却站着 `<kbd>综合评分 -12</kbd>`。

**P9 方案的核心目标**：**绝对禁止在业务层粗暴修改/翻转底层的真实物理信号**，转而通过 UI 分组大表头（Grouped Headers）实现“双轨参照系”同框对账，让“盘口大赢面”与“现货短线抛压”达成逻辑和视觉上的完满闭环。

## 🛠️ 具体实施架构 (Architecture & UI Specs)

### 1. 【上半区】多时间框架预测 (纯现货指标看板)
彻底剔除 Polymarket 基准价的数据污染，全部对齐实时现货口径。

* **列名净化**：
  * `基准价` 字段名修改为 `现货基准价`，确保所有周期（5M 到 1D）此处填入的都是当下瞬时现货价 `$71,402`。
  * `涨跌幅` 字段名修改为 `现货涨跌幅`，严谨表达“引擎目标价相对当前现货价的涨跌”（如 `-0.1%`）。
* **数据血统**：引擎算什么方向，就原汁原味展露什么方向。这里是交易员最硬核的技术内参。

### 2. 【下半区】Polymarket 投注指南 (双轨制看板)
引入 HTML `<colgroup>` 和 `<th colspan="X">` 分组机制，在视觉层面像楚河汉界一样将桌子一分为二。既保留您的原生分数，又附加专门的解说词。

#### 功能列释义：
1. **左半场 (PM 视角)**：专注告诉投注者**底牌和安全感**。只要目标价大于 PM死线，盈亏比就是耀眼的绿色。当盈亏比巨大时，`PM 原初建议` 智能输出“稳赢局”或“强势看涨”。
2. **右半场 (现货视角)**：以红蓝分明的数值直击**现货短期压力**。新增的 `现货实时价格` 锚点列发挥奇效——用户左眼看到 `现货 $71,402`，右眼看到 `目标 $71,333`，秒懂此时“正在下跌”，瞬间合理化了后边的 `-26` 分。
3. **列替换**：原 `方向状态` （如同向增强/方向矛盾）在赌盘场景中造成巨大的词不达意，升级为 `现货抛压状态`。由后端输出“短线回调压力”、“上升动能分歧”、“下穿盘口死线”等人话解说。

## 📌 备忘录 (Memo): 六大因子评分的极度内敛约束

> 这是一个已确认的算法特征，现阶段保持现状，**暂不修改代码**，特此记录以备后续优化体系时参考。

**现象**：前端长年显示的“因子评分”极小（通常在 -5 到 +5 之间徘徊，极少突破 ±10），虽看似 10 分制，实则底层量纲确为 **±100** 坐标系。

**根本原因：多因子苛刻对冲算法**
1. **单项限流**：每项因子（MACD, 布林带, KDJ 等）得分硬性封顶/封底在 `[-3, +3]` 区间内。
2. **严苛分母**：所有分数加和后，会被除以 `(总体权重 × 3)` 强行逼近 `[-1.0, +1.0]` 的极小区间。
3. **极强中和**：传统指标常年面临“多空撕裂”（如 MACD 看涨，KDJ 却超买看跌）。在上述苛刻的加权平均算法下，正负极易互相抵消。
4. **结论**：要想拿到 ±100 满分，必须所有 6 个指标在同一秒钟达到绝对巅峰。这导致日常震荡市中，因子如同高度阻尼的弹簧，分数被紧紧压缩在 0 轴附近，沦为纯粹的“防守型滤网”而缺乏激进的方向指示器作用。

**后续优化设想（待定）**：
若未来希望因子发挥更大、更敏锐的引导作用，可考虑（1）削减除数阻尼；（2）引入非线性放大器（如当三个主力因子同向时，分数呈指数级放大突破 0 轴泥潭）。

## 🗂️ 涉及的变动与架构影响评估 (Impact Assessment)

为了实现 P9 的双轨制，系统架构需要在表现层和业务层做轻量手术，**完全不碰底层计算核心（chan.py / scoring.py）**。

### 1. 前端变动 (Frontend)
* **文件**：`frontend/src/components/PolymarketGuide.tsx`, `PredictionTable.tsx`
* **UI 框架结构重组**：原生支持 `<colgroup>` 的 Table 重构。这标志着表格从“单行线性”升级为“复合层级结构”。
* **响应式升级**：表头增多（加至 10 列），对 `overflow-x-auto` 和 Tailwind CSS 的最小列宽要求变得极高。

### 2. 后端变动 (Backend)
* **文件**：`backend/app/services/polymarket_service.py` (`_generate_guides` 方法)
* **API 结构扩充**：响应 JSON 新增以下两个字段，前端直接渲染，后端全包办处理逻辑：
  * `pm_action_advice`: 综合盘口安全垫算出的 Polymarket 建议文案（例：`↑ 稳赢局 (做多)`）
  * `spot_momentum_desc`: 解析为什么信号常年为负的翻译文案（例：`短线回调压力`）

## ⚠️ 更新注意事项与潜在风险 (Risks & Attention Points)

1. **移动端横向溢出风险**：双参照系带来了 10 列庞大数据。虽然在宽屏显示器上极其震撼，但在移动设备上必须确保横向滚动条的丝滑，不可强制换行破坏表头对齐。
2. **WebSocket 与轮询的时间差错位**：PM 盘口价格是通过 WebSocket 推送的，而引擎计算是靠定时轮询的。如果在现货秒级暴跌的瞬间，两者的数据帧没有对齐，可能会出现“安全垫”短暂计算错误的情况。目前依靠 5 分钟轮询覆盖已足够安全，但需在极端行情下长效观察。
3. **PM原初建议“过度自信”的陷阱**：盘口安全垫算法不能仅仅看 `> 0`。如果距 PM开盘价只有 `+0.01%`，建议不可直接出“稳赢局”，必须引入**阈值分级机制**（如 `>1.5%` 为稳赢，否则综合考量现货分数）。
