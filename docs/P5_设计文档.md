# 实施计划: Polymarket 短线盘口 Chainlink 基准价提取 (P5)

## 背景问题 (Bug 核心)
在解决 Polymarket 短期滚动盘口（5m/15m/4h）数据拉取时发现：官方的 `/markets/slug` Gamma API 虽然公开返回了实时涨跌胜率 (`yesPrice`/`noPrice`)，却**没有下发用来做基准判定的价格 (Price To Beat)**。
由于 Polymarket 官方结算依托于 Chainlink Oracle（而不是单纯的币安现货），微小的价差（Spread）在此类几十美金定胜负的极短线盘口中可能导致致命误判。因此，绝对不能在后端使用现价（Current Price）或币安开盘价进行容错平替。

---

## 最终选定方案：官方 RTDS WebSocket 实时订阅预言机 (最佳解法)

经过 USER 最高级别情报验证，Polymarket 存在唯一完美获取真实 Chainlink 结算价格的通道，该通道与官方前端 UI 及结算合约 100% 保持一致，彻底杜绝 Binance 现货价差问题：

**核心接口**：`wss://ws-live-data.polymarket.com` (官方 RTDS WebSocket)
**订阅主题**：`crypto_prices_chainlink` (`btc/usd`)

### 核心实施流
1. **构建常驻守护线程**：在 FastAPI 后端启动时，通过 `websockets` 或 `websocket-client` 库建立一条永不挂断的后台守护线程，专门监听该 WSS 推送。
2. **基准点精准捕获**：
   - 维护一个字典，保存当前所有正在关切的时间框架（如 5m, 15m, 4h）的 `start_ts`（例如 09:30:00 的秒级时间戳）。
   - 一旦 WebSocket 广播的最新的 Chainlink 实盘价格到达，将其所带的系统时间与 `start_ts` 做极小误差判断（如 `abs(now - start_ts) <= 3` 秒）。
   - 精准在这个瞬间，把该价格死死锚定（Latch），直接作为该时间框架的 `priceToBeat`。
3. **数据交接库**：将捕获到的官方基准价通过后端 Cache 锁或者 Redis 共享给每次 API 请求。
4. **断线重连容错**：由于 WSS 可能存在断线，需要增加 [on_close](file:///d:/AI_Projects/project%20%282%29/backend/test_ws.py#26-28) 的重连保护，并在错过 3 秒窗口时的极小概率情况下，退而求其次通过 REST 或缓存上一个最近值兜底。

该方案将完全废弃此前引发超时的 API 猜测逻辑，用纯天然的官方监听解决一切基准漂移问题！

## 核心扩充：全链路时间绝对对齐与节拍同步 (Time Synchronization)

为彻底消灭“因抢跑或滞后导致的看对图表输掉结算”的问题，系统必须全面升级为**事件节拍驱动（Event-Driven Tick）**模型，达成前端、后端、预言机三者的毫秒级共振：

### 1. 基准准星锁定 (Base Price Latch)
- 完全依赖上文确立的 RTDS WebSocket。废弃主动猜测倒计时，仅在预言机**真实下发**印有 `start_ts` 戳记的那一包数据时，瞬间锁死当前周期的基准价。物理上实现 0 毫米误差开盘对齐。

### 2. 后端心脏起搏器 (Backend Heartbeat Tick)
- **废除请求驱动**：后端 API 不再因前端的刷新频率 (`setInterval`) 被动触发计算。
- **强制时钟对齐**：后端设立全局任务循环，严格对齐原子钟 (NTP)。
  - 当时间步入周期切换点（如 1 分钟、5 分钟的 `00 秒`）。
  - 后端强制挂起等待至 `02 秒`（容忍 Binance K线成型收敛的物理网络延迟）。
  - 在 `02 秒` 瞬间并发拉取锁死的 K 线交由缠论引擎计算，并将产生的评分硬写入内部快照 (Atomic Cache Swap)。
- **前端幽灵隔离**：前端的不规则查询，永远只能读取这块刚落地的快照，彻底剥离查询时间点对计算上下文的污染。

### 3. 结算死水区封锁 (Dead-Zone Lockdown)
- 在任何短周期盘口临近决战结算前（例如 15 分钟盘口的最后 1 分钟，或 5 分钟盘口的最后 30 秒），由于预言机节点聚合、交易流动性干涸，引擎进入“信号静默期 (Dead-zone)”。强制保持之前发出的持仓信号，拒绝生成由于微小跳动引起的翻转错觉。

## 时间同步方案的系统隐患审查 (Hidden Dangers)

执行“心脏起搏器”同步机制将极大提升系统严谨度，但也引入了以下维度的隐患，需在代码实施时重点设防：

### 隐患 A：本地系统时钟发生 NTP 漂移 (Clock Drift)
我们要卡在 `00 秒` 发起节拍，如果服务器操作系统的本地时钟比真实世界（Polygon 区块链/Binance 服务器）慢了 2 秒。那么我们的 `00` 秒其实是别人的 `02` 秒，如果再加上 2 秒等待，去拉取 K 线时，可能刚好拿到了一根“还差 1 秒才收线”的未闭合 K 线，导致最后一根 K 线的技术指标极其抖动。  
*(对策：在每次启动前后，可以请求一次 `GET api.binance.com/api/v3/time` 校对微秒级偏移量 (Offset)，将该 Offset 叠加在本地判断逻辑中。)*

### 隐患 B：02 秒高并发引发的 API 限流断链 (Rate Limit Spike)
当我们把所有的拉取动作全挤压在全世界都在抢夺 K 线切分的 `02 秒`，如果 Binance 稍微卡顿，或者我们并发拉取 8 大周期 K 线的链接池被封锁拦截，整个周期的“打分节拍”就会因报错而流产，导致接下来长达 5 分钟内前端拿不到任何更新。  
*(对策：必须在协程内包裹重试与降级机制。若 `02 秒` 失败，立刻退避至 `03 秒` 盲目重试。保证计算主函数的健壮性，绝不能让引擎在这一秒宕机崩溃。)*

### 隐患 C：运算耗时导致的“真空期” (Compute Latency)
如果拉取数据花了 1 秒，缠论引擎跑完 8 个周期矩阵花了 2.5 秒，最终结果诞生于 `05.5 秒`。如果在这真空的几秒钟内，前端恰好发起了 API 请求，会遭遇等待或者取到陈旧脏数据。  
*(对策：采用原子指针替换 (Atomic Pointer Swap)。计算只在后台进行，而在计算彻底完成前，前端请求无感地命中上一帧缓存；唯有新矩阵落库瞬间，缓存锁指针才切向新数据，实现前端无缝平滑衔接。)*
